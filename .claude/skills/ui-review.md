# /ui-review — UI/UX 审查

对 UI 设计稿或页面截图进行专业审查，输出可直接执行原子化修复的 SPEC 文档。

## 模型约束

**强制使用 `gemini-2.5-flash`**，禁止调用 Pro 模型。

> 如果你认为 flash 能力不足，请输出 `⚠️ 建议人工复核`，而不是升级模型。

---

## 两条路径

| 路径 | 触发条件 | 执行流程 |
|------|---------|---------|
| **路径 A：单页审查** | 用户指定单个页面 | MiniMax 自检 → Gemini 审查 |
| **路径 B：批量审查** | 用户有 N 个相似页面（例：20 个列表页） | MiniMax 先修"模板页" → Gemini 审计模板 → MiniMax 批量套用 |

---

## 路径 A：单页审查

### Step 1: MiniMax 自检（新增）

使用 `minimax_generate_image` 或直接审查截图前，MiniMax 先读取现有代码进行自检。

**MiniMax 自检 Prompt**：
```
你是高级前端开发者。请在提交给 Gemini 审查之前，先自检代码。

## 当前代码
{读取目标文件内容}

## 自检清单
1. [ ] 是否使用了 project 的 Tailwind 规范（检查 tailwind.config.ts 的 theme.extend.colors）
2. [ ] 是否存在明显的对比度问题（灰色文字 on 白色背景）
3. [ ] 组件是否遵循现有项目的组件结构模式
4. [ ] 是否引入了新的状态而未处理（loading/error/empty）

## 输出
如果自检发现问题，在以下格式中列出：

### 自检问题
| 文件 | 行号 | 问题 | 自检修复建议 |
|------|------|------|------------|
| xxx.tsx | 42 | text-gray-300 对比度不足 | text-gray-600 |

如果自检通过：
### 自检结论
✅ 自检通过，无需修复建议，可直接提交 Gemini 审查。
```

> **强制要求**：MiniMax 必须完成自检并输出结论后，才能进入下一步。未进行自检的代码不得直接提交 Gemini 审查。

---

### Step 2: Validation — 对齐检查（设计稿场景）

如果用户提供的是设计稿（非现有代码），先验证设计稿是否包含所有必要组件。

**Validation Prompt**：
```
你是需求核对员。请检查以下设计稿是否包含了用户要求的所有功能。

## 用户原始需求（必须包含）
{用户Step 1中确认的组件列表}

## 设计稿
![设计稿](file://{图片路径})

## 检查结果

请逐项核对：
- [ ] {功能点1}：✅ 包含 / ❌ 缺失

## 结论
- ✅ **通过** — 所有必要组件均已包含，进入正式审查。
- ❌ **未通过** — 以下功能点缺失，请标记并要求重新生成：
  - 缺失功能点列表
```

> 如果 Validation 未通过，直接返回用户要求重新生成，不要进入 Step 3。

---

### Step 3: Gemini 审查（约束：只审计增量，拒绝冗余）

使用 `gemini_chat` MCP 工具，**强制 `model: gemini-2.5-flash`**。

**审查 Prompt（约束版）**：
```
你是 UI/UX 审查专家。请审查以下代码变更。

## 约束（强制执行）
1. **只审计增量** — 不要重复审计未修改的部分
2. **拒绝冗余说明** — 每个问题只描述一次，不要分散在多处说明
3. **合并全局建议** — 同样的根因问题只输出一个建议，不要按文件重复输出
4. **严格评分** — 10分制，8分以下才需要列问题

## 代码变更
```diff
{git diff 或 MiniMax 自检后的问题清单}
```

## 项目背景
- 页面类型：{页面类型}
- Tailwind 规范：{读取 tailwind.config.ts 中的 theme.extend.colors}

---

## 审查输出

### 评分（仅当 < 8分时详细说明）

| 维度 | 评分 | 问题数 |
|------|------|--------|
| 视觉一致性 | X/10 | N |
| 布局结构 | X/10 | N |
| 交互可用性 | X/10 | N |
| 响应式适配 | X/10 | N |
| 可访问性 | X/10 | N |

### 问题清单（合并去重版）

按**根因**合并问题，不要按文件重复：

#### 🔴 P0（阻塞，必须修复）

| 根因 | 影响文件 | 当前代码 | 修复代码 | 修复原因 |
|------|---------|---------|---------|---------|
| 对比度不足 | page.tsx, Card.tsx | text-gray-300 | text-gray-600 | WCAG AA 要求 ≥ 4.5:1 |

#### 🟡 P1（建议修复）

| 根因 | 影响文件 | 当前代码 | 修复代码 | 修复原因 |
|------|---------|---------|---------|---------|
| 间距不统一 | 所有页面 | p-4 | p-6 | 4px 基准网格 |

### 视觉资产提取

从设计稿或代码中提取 Tailwind 色值：

| 用途 | 当前色值 | 建议色值 | Tailwind 变量 |
|------|---------|---------|--------------|
| 主色 | 无 | #XXXXXX | primary |
| 成功色 | 无 | #XXXXXX | success |

---

### 🔧 技术修复规格 (Target: MiniMax/Claude Code)

| 目标文件 | 当前状态 | 修复代码/类名 | 根因 | 优先级 |
| :--- | :--- | :--- | :--- | :--- |
| `src/app/page.tsx` | `text-gray-300` | `text-gray-600` | 对比度不足 | P0 |
| `tailwind.config.ts` | 无 | `primary: '#XXXXXX'` | 色彩系统 | P1 |

### Tailwind 配置修改

```diff
# tailwind.config.ts
  theme: {
    extend: {
+     colors: {
+       primary: '#XXXXXX',
+     }
    }
  }
```

---

## 审查结论

**是否通过**：✅ 通过 / ❌ 需要修改后重审
**P0 阻塞问题数**：N（必须修复）
**Gemini 审查Token节省**：N%（相比冗余输出）
```

## 路径 B：批量审查

当用户有 N 个相似页面时，采用「模板页 → 审计 → 批量复用」流程。

### Step 1: 用户指定或选择「模板页」

用户从 N 个页面中指定 1 个作为「模板页」。

---

### Step 2: MiniMax 修复模板页

MiniMax 对模板页进行修复（遵循路径 A 的自检流程）。

---

### Step 3: Gemini 审计模板页，产出通用准则

使用 `gemini_chat`（强制 `gemini-2.5-flash`），审查模板页并产出 **UI_FIX_SPECS.md**。

**UI_FIX_SPECS.md 内容模板**：
```
# UI_FIX_SPECS — {页面类型} 通用修复准则

## 适用范围
本准则适用于所有 `{页面类型}` 页面。

## 通用问题清单（全局有效）

### P0 问题（所有页面必须修复）

| 问题类型 | 根因 | 修复方式 | 验证方法 |
|---------|------|---------|---------|
| 对比度不足 | text-gray-300 on 白底 | → text-gray-600 | WCAG AA 检测 |
| 主色缺失 | 无 primary | → theme.extend.colors.primary | 视觉一致性检查 |

### P1 问题（建议修复）

| 问题类型 | 根因 | 修复方式 |
|---------|------|---------|
| 间距不统一 | 混合 p-4/p-6 | → 统一 p-6 |
| 缺少加载态 | 无 skeleton | → 添加 skeleton 组件 |

## Tailwind 通用配置

```diff
# tailwind.config.ts
  theme: {
    extend: {
      colors: {
+       primary: '#XXXXXX',
+       success: '#XXXXXX',
+       warning: '#XXXXXX',
+       danger: '#XXXXXX',
      },
      spacing: {
+       '18': '4.5rem',
      }
    }
  }
```

## 批量验证清单

修复完成后，请逐项验证：
- [ ] 对比度：所有 text-gray-300 → text-gray-600
- [ ] 主色：所有页面引用 tailwind.config.ts 中的 primary
- [ ] 间距：所有 padding 统一为 p-6
- [ ] 加载态：列表页均包含 skeleton

## 增量 Diff 审计

每次 MiniMax 批量修复后，输出以下格式的 diff 自检报告：

```
## Diff 自检报告

### 修改文件
- `src/pages/Page1.tsx`
- `src/pages/Page2.tsx`
- ...

### P0 修复验证
- [ ] 所有 text-gray-300 → text-gray-600 ✅/❌
- [ ] 所有页面已应用 primary 色值 ✅/❌

### P1 修复验证
- [ ] 所有页面间距统一为 p-6 ✅/❌
- [ ] 列表页加载态已添加 ✅/❌

### 结论
- ✅ 全部通过，可进入终审
- ⚠️ 仍有问题，请修复后再审
```
```

### Step 4: MiniMax 批量套用

MiniMax 读取 `UI_FIX_SPECS.md`，对剩余 N-1 个页面批量套用修复。

**MiniMax 批量执行 Prompt**：
```
你是前端开发工程师。请根据以下通用准则，批量修复 {N-1} 个页面。

## 通用准则
读取：`docs/ui-review/UI_FIX_SPECS.md`

## 需要修复的页面列表
{列出剩余 N-1 个页面路径}

## 执行要求
1. 逐个页面修复，遵循 UI_FIX_SPECS.md 中的修复方式
2. 每个页面修复后，输出一个简短的 diff 自检报告
3. 所有页面修复完成后，输出汇总报告

## 输出格式
### 批量修复汇总

| 页面 | 修复问题数 | P0 | P1 | 状态 |
|------|-----------|----|----|------|
| Page1.tsx | 3 | 1 | 2 | ✅ |
| Page2.tsx | 2 | 0 | 2 | ✅ |
"""
```

---

### Step 5: 终审截图（仅 P0 逻辑改动时触发）

**触发条件**：只有当存在 P0 级别的逻辑改动（如新增组件、改变布局结构）时，才触发终审截图。

**不触发终审的场景**（微调，直接自检即可）：
- 颜色调整（text-gray-300 → text-gray-600）
- 间距调整（p-4 → p-6）
- 类名替换（bg-blue-400 → bg-blue-600）

**终审流程**：

1. MiniMax 修复完成后，输出 diff
2. Claude 通过 `mcp__minimax__text_to_image` 生成终审截图（或用户手动提供截图）
3. Gemini Flash 审查截图，对比 `UI_FIX_SPECS.md` 验证

**终审 Prompt**：
```
你是终审审查专家。请验证批量修复是否符合通用准则。

## 通用准则
{UI_FIX_SPECS.md 核心内容}

## 终审截图
![终审截图](file://{图片路径})

## 验证结果
- [ ] P0 问题全部修复 ✅/❌
- [ ] P1 问题全部修复 ✅/❌
- [ ] 视觉一致性符合准则 ✅/❌

## 终审结论
✅ **通过** — 批量修复符合准则，审查结束。
❌ **未通过** — 以下问题需要修复：
- 问题列表
```

## 断点续传指令

### /continue-ui-review {页面名}

读取 `docs/ui-review/{页面名}_NEXT_ACTION.md`，继续执行。

**执行逻辑**：

1. 读取 `NEXT_ACTION.md`，检查已完成项
2. 对未完成项，调用 MiniMax 执行修复
3. **中间过程**：MiniMax 通过 `git diff` 输出 diff → Claude 读取 diff 文本自检（**不生图**）
4. 只有当 P0 级别逻辑改动完成时，才触发终审截图
5. 更新 `NEXT_ACTION.md`（打勾已完成项）
6. 如果所有 P0/P1 完成且无需终审 → 输出 `✅ 审查通过`，清理 `NEXT_ACTION.md`

## 约束汇总

| 约束 | 说明 |
|------|------|
| 禁止 Pro 模型 | 强制 gemini-2.5-flash |
| MiniMax 自检前置 | 未自检不得提交 Gemini |
| Gemini 只审增量 | 不重复审计未修改部分 |
| 问题合并去重 | 同根因只输出一个建议 |
| 批量复用准则 | UI_FIX_SPECS.md 作为通用准则 |
| 终审按需触发 | 微调不生图，P0 逻辑改动才终审 |

---

## 注意事项

- Validation 未通过时，必须重新生成，不要跳过
- MiniMax 自检是强制步骤，不能省略
- Gemini 审查时每个问题只输出一次，不要重复
- 批量场景下，UI_FIX_SPECS.md 是核心工件，必须完整输出
- 终审截图仅在 P0 逻辑改动时触发，微调直接 diff 自检
