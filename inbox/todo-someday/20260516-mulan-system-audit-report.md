# Mulan BI Platform 系统定位与架构审计报告

**审计时间**：2026-05-16
**联合署名**：Gemini 1.5 Pro (CLI Agent)、Minimax (Architect Review)、GPT-5 Codex (Data QA Review)
**审计目标**：评估系统架构与当前定位是否能支撑”10~2000 人企业”、”千人千问/千人千面”、”自然语言 AI 数据消费体验”以及相关的并发与安全要求。
**约束声明**：本报告基于静态代码与文档审查，不包含真实压测、渗透测试或生产流量审计结论。涉及并发、QPS、缓存命中率等容量数字，若无明确实现或压测证据，均按风险假设处理。

---

## 1. 核心定位与量化阐述审查

### 审计结论：顶层定位不匹配，系统级量化目标缺失。

*   **文档现状**：当前的 `README.md` 首行将系统定位为：“*数据建模与治理平台 — 面向 BI 运维团队的数据质量、DDL 规范、Tableau 资产治理、语义维护工具。当前阶段：v0.x，内部 dogfooding。*”
*   **缺失项**：
    *   顶层 `README.md` 没有提及“适配 10~2000 人的公司”这一量化目标。
    *   顶层系统叙事没有体现“千人千问”、“自然语言消费数据”的最终用户价值。
    *   顶层定位没有体现“AI Powered”的平台核心调性，目前的表述更像是一个中后台的运维面板。
*   **补充说明**：局部规格文档已经出现问数、Data Agent、首页数据问答、Data QA Golden Set 等能力设计，但这些能力尚未上升为系统 README 与总体架构蓝图中的统一产品定位。

### 🚀 改进 Action (PM/架构师)
1.  **重写系统愿景**：更新 `README.md` 和产品架构蓝图，将定位从“面向 BI 运维的内部工具”升级为“企业级 AI 驱动的数据消费平台”。
2.  **明确量化指标**：在文档中明确声明支持 2000 人级别的用户规模、50+ 并发问答、以及 >95% 的严格语义准确率目标。

## 2. 千人千面与扩展性审查

### 审计结论：架构重“确定性与防守”，目前属于“受控架构”，尚未形成体系化的个性化扩展能力。

*   **架构现状**：基于 `docs/specs/36` 和 `56` 的架构约束，Mulan 采用了极度收敛的控制流：
    *   **Intent 路由受限**：根据 `backend/services/data_agent/intent_classifier.py` 的定义，自然语言提问在入口处被强制正则路由到 8 种 `DataAgentIntent`（如 `aggregate`、`ranking`、`set_difference` 等）。虽然 Spec 36 预留了 `ToolRegistry` 作为扩展机制，但当前核心问数链路高度依赖这 8 个静态入口意图进行分发。这在保障业务确定性的同时，限制了 LLM 应对高度复杂、发散的探索性分析的能力。
    *   **计算下推与扩展性取舍**：系统严格冻结了 Python 层的动态派生计算，强制依赖前置的 `Metrics Registry` 和 Tableau 的基础聚合。这是一个**为了保障口径一致性 (SSOT) 的主动设计选择**，并非架构缺陷。但客观上，在 Metrics Registry 具备极强动态扩展能力之前，它限制了用户随时“捏”一个临时指标的个性化体验。
*   **千人千面缺口**：目前系统主要支持基于角色、用户组、Tableau RLS 和会话上下文的差异化访问，尚未形成产品层面的“千人千面”个性化体验。“千人千面”缺口的核心根源在于：
    1.  **缺乏用户画像与上下文感知**：当前的路由逻辑（如 `route_datasource()`）和规划器没有集成对用户历史习惯、关注指标的长期记忆（Memory）。
    2.  *(注：之前的表述中将 Guardrail 视为了个性化阻碍，这并不准确。`mcp_args_guardrail` 和 Result Guardrail 是安全与质量的底线防线，个性化的缺失应当归因于路由层和规划层缺少用户画像数据输入。)*

### 🚀 改进 Action (架构师)
1.  **从静态 Intent 走向动态 Task Plan**：随着底层 `ToolRegistry` 的成熟，逐步放宽正则分类器的强拦截，将单一 `Intent` 升级为基于大模型的“多步任务动态编排 (Task Plan)”，允许在安全沙箱内组合更复杂的数据处理流。
2.  **引入用户级 Semantic Memory**：在问答上下文中注入基于向量库的用户专属 Memory（记录常用的过滤维度、偏好指标），实现真正的“千人千问”解读。

## 3. 并发登录与问答支撑审查 (50+人并发)

### 审计结论：应用层具备并发处理基础，但尚无证据证明可稳定支撑 50+ 人并发问答；底层存在多层容量瓶颈。

*   **支持项**：后端基于 `FastAPI`，前端配合 React Query，具备并发 HTTP 请求处理基础。部分 MCP/Tableau 上下文使用了 `contextvars`，有助于降低多用户并发请求导致上下文串签的风险。
*   **瓶颈与风险**：
    *   **数据库连接池上限**：当前默认 PostgreSQL 连接池配置为 `pool_size=10` + `max_overflow=20`，总上限约 30 连接，可通过 `DB_POOL_SIZE` / `DB_MAX_OVERFLOW` 环境变量调整。以 50 人并发问答估算，若每个 SSE 问答链路持续占用 DB session 或伴随后台任务写入，容易触发连接等待。建议在架构文档中标注压测数据，并设计连接池饱和时的排队降级策略。
    *   **同步 DB 与长连接风险**：当前数据库层使用同步 SQLAlchemy engine；首页问答又包含 SSE、LLM、MCP/Tableau 等长耗时链路。即使 Web 框架支持 async，也不能直接推导出 50+ 并发问答能力。
    *   **Tableau Server / MCP 瓶颈**：所有的高并发问数最终都会穿透到底层 Tableau Server / MCP。当前未见针对 50+ 并发问答的真实压测数据；Tableau/MCP 的实际 QPS 上限需以压测确认。50 人并发存在请求排队、连接耗尽和雪崩风险。
    *   **大模型速率限制**：如果 LLM Provider 端没有做充沛的并发配额（Rate Limit），系统会大面积触发 429 错误。

### 🚀 改进 Action (研发/运维)
1.  **引入高并发熔断与排队机制**：在 `MCPToolExecutor` 执行层增加 Circuit Breaker；针对复杂查询引入基于 Celery/Redis 的异步任务排队和轮询机制。
2.  **语义缓存 (Semantic Cache)**：引入基于向量相似度的 NLQ 缓存（针对重复的高频宏观问题），降低穿透到 Tableau 和大模型的压力；缓存命中率必须由真实问题分布验证，不能预设固定收益。
3.  **数据库连接池监控与扩容**：增加连接池健康度监控告警；生产环境根据实际压测结果考虑横向扩容或连接池调优。

## 4. 2000 人企业级系统安全设计审查

### 审计结论：基础安全能力较扎实，分层防护已有基础；但距离 2000 人企业采购级安全与合规仍有明显缺口。

#### 4.1 已有企业级安全能力（分层防线，已验证）

| 能力 | 实现状态 | 说明 |
|------|---------|------|
| 密码哈希 | ✅ 已实现 | PBKDF2-SHA256，100k 迭代 |
| JWT 会话 | ✅ 已实现 | HTTP-only Cookie + 7 天过期 + Refresh Token Sliding Window |
| MFA TOTP | ✅ 已实现 | Sprint 3 完成，用户可启用/禁用 |
| RBAC 四级角色 | ✅ 已实现 | admin > data_admin > analyst > user |
| 密钥隔离 | ✅ 已实现 | 4 把密钥分离（Fernet），SESSION/DATASOURCE/TABLEAU/LLM |
| XSS 防护 | ✅ 已实现 | HTML 转义渲染 |
| SQL 注入防护 | ✅ 已实现 | 参数化查询（SQLAlchemy ORM） |
| 注册限流 | ✅ 已实现 | IP 级别 60s 内 5 次 |
| 行级权限 (RLS) | ✅ 已实现 | Tableau Connected Apps JWT 注入，继承 Tableau RLS |

#### 4.2 企业级安全缺口（需新增）

| 能力 | 期望 | 现状 | 风险级别 |
|------|------|------|---------|
| 统一身份认证 (SSO) | OIDC/SAML 2.0 | 仅本地密码校验 | 🔴 高 |
| 完整审计日志 | 全量操作审计 | 局部 operation log / Trace / MCP query log 存在或已有规划，但统一审计 API 仍为 placeholder，缺乏合规审计视图 | 🔴 高 |
| 密码策略 | 复杂度/过期/历史 | 未实现 | 🟡 中 |
| 细粒度权限 | `tableau:read` vs `tableau:write` | 当前粗粒度 8 项权限键 | 🟡 中 |
| 会话治理 | 强制登出/会话上限 | 无会话上限控制 | 🟡 中 |
| 多租户隔离 | 租户字段 + 数据隔离 | 系统不是 SaaS 多租户架构；部分分析链路预留 `tenant_id`，但认证、用户、连接、首页问数和审计未形成端到端租户隔离 | 🟡 中 |
| 敏感数据脱敏 | 日志脱敏 | 密码/密钥已防护，其他敏感字段未描述 | 🟡 中 |

#### 4.3 安全缺口影响分析

**SSO 缺失** 是 2000 人企业采购的最大障碍。大型企业不会接受本地密码管理，要求与 AD 域/飞书/钉钉等企业身份提供商集成。

**统一审计闭环不足** 影响合规采购。金融/医疗等监管行业需要”谁在什么时间查了什么数据”的完整审计追溯。当前系统已有局部日志与 trace，但尚未形成覆盖所有数据流出节点、可检索、可导出、可留存治理的统一审计能力。

### 🚀 改进 Action (PM/研发)
1.  **集成企业 SSO**：规划 `OIDC` / `SAML 2.0` 登录模块，支持与企业 AD 域、飞书、钉钉无缝对接。
2.  **建设全局审计日志中台**：统一拦截所有数据流出节点，记录时间、人员、敏感字段命中情况，提供专门的安全审计看板以满足中大型企业的采购合规要求。
3.  **细化 RBAC 权限模型**：将粗粒度 8 项权限键升级为细粒度资源级权限（如 `tableau:read` / `tableau:write`）。
4.  **增加会话治理**：实现管理员强制登出、会话上限控制。
5.  **明确部署模型与租户边界**：若定位为单企业私有化部署，应在文档中明确“非 SaaS 多租户”；若未来支持多企业托管，应补齐端到端 `tenant_id`、数据隔离、审计隔离和运维隔离设计。
