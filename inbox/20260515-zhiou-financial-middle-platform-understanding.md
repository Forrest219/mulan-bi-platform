# 致欧家居财务中台项目 - 需求理解与联合解决方案分析

> **编写角色**：Product Manager
> **输出日期**：2026-05-15
> **核心策略**：采用 **Mulan BI Platform (数据治理与智能中枢) + Tableau (可视化与分析引擎)** 的联合解决方案，避免重复造轮子，最大化发挥各自优势。

---

## 1. 客户核心痛点与诉求提炼

通过对《财务中台项目-业务需求说明书》的深入剖析，客户（致欧家居）当前面临的核心痛点及建设诉求可归纳为以下四个维度：

1. **数据孤岛与口径割裂**：OMS、SAP、物流等十余个业务系统数据缺乏统一标准，导致财务核算与管理口径不一致。
2. **人工核对与操作成本极高**：系统间数据对账、会计凭证合并、海外税务申报均重度依赖人工，效率低且易出错。
3. **分析滞后与缺乏全局视角**：“三表合一”（毛利监控、业务毛利、管报毛利）难以拉通，缺乏多维度、实时的管理报表和月结监控体系。
4. **智能化期望**：期望业务人员能够通过低代码/拖拉拽或 AI 辅助（财务自主分析平台）实现数据的自助清洗、整合与洞察。

## 2. “Mulan BI + Tableau” 联合解决方案整体架构

为了满足客户需求，同时坚守系统边界，我们提出“专业工具做专业事”的联合架构：

*   **数据接入与存储层（需基础设施支持）**：企业级数仓（如 PostgreSQL/StarRocks）+ 专业 ETL/ELT 数据集成工具，负责将多业务系统数据物理搬运入仓。
*   **数据治理与智能层（Mulan BI Platform 承接）**：作为整个中台的“大脑”，提供主数据定义、合规扫描、DQC 数据质量核对调度、统一语义指标管理，以及 Data Agent/SQL Agent 的 AI 问数能力。
*   **展现与可视化层（Tableau 承接）**：作为“表现层”，负责所有管理报表、多维钻取、月结驾驶舱及海外税务看板的呈现。

## 3. 核心需求拆解与系统承接映射

基于联合方案，对《业务需求说明书》中的各项核心需求进行落位分配：

### 3.1 财务数据仓库与数据治理
*   **客户需求**：多系统数据集成、主数据管理、数据质量监控。
*   **承接方案**：
    *   **Mulan BI** 承接 **数据治理与质量监控**：利用平台原生的“数仓健康扫描”与“DDL 合规检查”能力，确保入仓的财务模型符合主数据规范（如客商、成本中心命名）。
    *   **Mulan BI** 承接 **语义/指标管理**：通过“指标管理模块”集中定义财务科目与映射规则，确保口径唯一。
    *   *(注：物理数据的 ETL 抽取转换过程，建议借助专门的数据集成中间件实现，Mulan BI 负责对抽取结果进行扫描与校验。)*

### 3.2 财务数据核对自动化
*   **客户需求**：规则配置可视化、自动触发对账、大批量处理、异常结果追踪。
*   **承接方案**：
    *   **Mulan BI** 承接：完全复用 Mulan BI 的 **DQC (数据质量监控) Pipeline**。财务人员可在 Mulan BI 前端可视化配置对账规则（如：OMS 金额 == SAP 金额），系统基于 Celery 定时调度大批量核对任务。
    *   **Tableau** 承接：核对结果与异常明细记录推送至数仓后，由 Tableau 制作“核对异常监控看板”供业务人员钻取追踪。

### 3.3 管理报表体系与月结驾驶舱
*   **客户需求**：“三表合一”、多维度下钻、报表版本管理、月结进度与异常预警、拖拽式报表开发。
*   **承接方案**：
    *   **Mulan BI** 承接：提供底层的“Tableau 健康评分”与“资产巡检”，确保发布的财报不卡顿、未过期；提供 **语义治理**，确保 Tableau 数据源中的字段中英翻译、毛利计算公式准确无误。
    *   **Tableau** 承接：利用 Tableau Desktop/Web Edit 的强大多维分析能力实现拖拽式分析、图表构建、邮件订阅推送以及月结流程驾驶舱。

### 3.4 财务自主分析平台（AI 应用）
*   **客户需求**：轻代码、自主清洗与数据洞察分析。
*   **承接方案**：
    *   **Mulan BI** 承接：通过平台的 **Data Agent** 和 **SQL Agent**，财务人员可以通过自然语言（NL-to-Query）直接询问诸如“上月欧洲区各渠道业务毛利对比”，Agent 结合知识库直接给出数据或生成图表，极大降低分析门槛。

### 3.5 会计凭证合并与海外税务管理
*   **客户需求**：按规则合并凭证后传输至 SAP、海外税务自动计算与过程留档。
*   **承接方案**：
    *   **Mulan BI / 业务规则引擎**：在 Mulan BI 内部开放扩展模块，或者结合外部 Python 脚本任务。由 Mulan BI 的任务调度系统（Task Management）定时触发预处理 SQL 或 Python 脚本，完成凭证合并计算和税额计算，并将过程与结果落库归档。
    *   接口交互：计算结果通过标准 API 或中间表的方式与 SAP 对接写入。

## 4. 前置风险与澄清点 (Risk & Constraints)

在向客户提案或进入技术设计前，需澄清以下边界：

1. **ETL 工具选型**：明确“财务中台”的数据抽取搬运（ETL）本身是否由客户现有工具（如 DataX、Kettle 等）承担。Mulan BI 主要负责“管”和“查”，不负责底层的“搬迁”。
2. **SAP 回写边界**：会计凭证合并后需要“传输至 SAP 实现凭证合并”。由于 SAP 的写入接口通常涉及极高的一致性要求和授权验证，建议采用“中间表 + 定时拉取”而非“中台强侵入式直写”的解耦方式。
3. **Tableau License**：联合方案重度依赖 Tableau 作为前端可视化工具，需确认致欧家居内部的 Tableau Server 授权和 Viewer 账号并发是否满足全员（特别是业务财务人员）的使用体量要求。
