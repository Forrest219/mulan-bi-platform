## 场景 1：LLM 名称唯一性

> 不允许同名 LLM 配置，前端、后端、数据库三层均须拦截。

| TC-ID | 用例名 | 前置条件 | 操作步骤 | 期望结果 | 当前状态 |
|-------|--------|----------|----------|----------|----------|
| LLM-1-01 | 新增重复显示名称被拦截 | 已存在 display_name="GPT-4o-prod" 的配置 | 在新建表单填写 display_name="GPT-4o-prod"，点击保存 | 前端即时红字提示"名称已存在"；后端若绕过前端仍返回 409 | ✅ 已修复 |
| LLM-1-02 | 编辑时改为已存在名称被拦截 | 已有 A、B 两条配置 | 编辑配置 A，将 display_name 改为配置 B 的名称，点击保存 | 前端提示重名；后端返回 409，配置 A 原名保留 | ✅ 已修复 |
| LLM-1-03 | 同名但不同 purpose 是否允许 | 已存在 display_name="GPT-4o"、purpose="default" | 新建 display_name="GPT-4o"、purpose="embedding" | 待产品确认：唯一性约束是否跨 purpose；当前行为需记录 | ⚠️ 待验证 |
| LLM-1-04 | display_name 为空时不做唯一性校验 | 无同名配置 | 新建两条 display_name 均为空的配置，分别保存 | 两条均保存成功，不触发唯一性报错（display_name 为可选字段） | ⚠️ 待验证 |

### 验收标准

- [ ] 前端表单提交前完成本地唯一性预检（debounce 查询或本地缓存对比）
- [ ] 后端数据库层存在 UNIQUE 约束或等效唯一索引
- [ ] 409 响应体包含可读错误信息，字段指向 display_name

## 维度 B：删除安全性（LLM-7-xx）

> 删除操作须防止破坏系统最低可用性；高危批量接口须有额外保护。

| TC-ID | 用例名 | 前置条件 | 操作步骤 | 期望结果 | 当前状态 |
|-------|--------|----------|----------|----------|----------|
| LLM-7-01 | 删除 default purpose 唯一 active 配置被拒绝 | default purpose 仅剩一条 is_active=true 配置 | DELETE /api/llm/configs/{id} | HTTP 400，提示"不能删除 default purpose 的唯一活跃配置" | ⚠️ 待验证 |
| LLM-7-02 | 删除非 default purpose 唯一 active 配置无保护 | embedding purpose 仅剩一条 is_active=true 配置 | DELETE /api/llm/configs/{id} | 当前返回 200 并删除（缺陷：应同样保护）| ❌ FAIL |
| LLM-7-03 | 删除后依赖该 purpose 的功能优雅降级 | 已删除某 purpose 的全部配置 | 触发依赖该 purpose 的功能（如 embedding） | 功能返回明确错误提示"该功能暂无可用 LLM 配置"，不抛出 500 | ❌ FAIL |
| LLM-7-04 | 旧接口 DELETE /config 批量删除无保护（高危） | 旧批量删除接口仍可用 | DELETE /api/llm/config（不含 id，批量） | 应返回 404（接口已下线）或 405（方法不允许）；当前接口若仍存在且无保护为高危缺陷 | ❌ FAIL |
| LLM-7-05 | 删除不存在的 config_id 返回 404 | 无 | DELETE /api/llm/configs/99999 | HTTP 404，响应体含"配置不存在" | ✅ PASS |

### 验收标准

- [ ] 删除前检查：若为某 purpose 最后一条 active 配置，统一拒绝（不区分 purpose 类型）
- [ ] 旧批量删除接口下线或加 admin 二次确认
- [ ] 功能模块调用 LLM 层时捕获"无可用配置"异常，返回 503 而非 500
