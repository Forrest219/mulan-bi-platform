## 一、问题陈述

### 1.1 RLS 失效风险

当前 Tableau MCP 使用运维人员的单一 PAT（Personal Access Token）执行所有查询。Tableau 的 Row-Level Security（RLS）是基于"当前登录用户身份"动态过滤数据的——当所有查询都以同一个运维账号身份发出时，RLS 规则形同虚设，任何用户都能拿到全量数据。

这意味着：
- 有销售大区隔离权限的数据，被所有人全量可见
- 有岗位或部门限制的敏感指标，无法隔离
- 平台实质上无法对普通 BI 用户开放

### 1.2 用户无法问数的现状

Mulan 当前面向运维和数据工程师提供管理界面。200+ 普通 BI 用户（业务分析师、运营、财务等）没有合适的入口：
- 进入 Mulan 后台会看到大量运维和配置功能，与他们无关
- 没有一个"只问数据"的干净窗口
- 即使能进入，查询结果不做用户隔离，不安全

### 1.3 已决定的技术方向

使用 **Tableau Connected Apps（JWT 用户模拟）** 解决 RLS 问题：Mulan 用 Connected App 密钥签发 JWT，携带目标用户的 Tableau 用户名，Tableau 据此以该用户身份执行查询，RLS 完整生效。

**已确认**：生产环境 Tableau Server 版本为 **2025.3.4**，完全支持 Connected Apps，JWT 用户模拟方案可直接落地，无需降级方案。

**已确认**：Mulan 与 Tableau 使用同一套 **AD/LDAP**，用户名完全一致，系统可自动透传用户身份，无需手工映射。

用户界面采用 **Mulan 内建"问数"前端模块**：在 Mulan 内部新建一个独立的"问数"前端页面，借鉴 Open WebUI 的 UX 理念（干净、优雅、聊天框为核心），面向 200+ 普通 BI 用户，与现有运维功能完全隔离。不部署独立的 Open WebUI 实例。

## 二、目标用户

**主要用户：普通 BI 用户**

典型角色包括：业务分析师、运营专员、区域销售经理、财务分析师。

画像特征：
- 熟悉业务指标，不懂 SQL、不懂 Tableau 后台配置
- 有数据访问权限，但权限是受限的（不同人看不同数据范围）
- 日常需求是"查某个指标、看某个维度的数据"，不是构建报表
- 习惯用自然语言表达问题，如"上个月华东区的销售额是多少"

**非目标用户（本需求不服务）：**
- 运维人员（使用 Mulan 管理后台）
- 数据工程师（使用 Mulan 语义治理功能）

## 三、功能范围（In Scope）

**核心链路：用户身份绑定 + 聊天式问数**

1. **用户身份自动透传**：Mulan 与 Tableau 共用同一套 AD/LDAP，用户名一致，登录 Mulan 后系统自动以当前用户的 AD 用户名作为 Tableau 用户名，无需手工映射。

2. **JWT 签发（Connected Apps）**：用户发起查询时，Mulan 后端以该用户的 Tableau 用户名签发 Connected App JWT，传递给 Tableau MCP 执行查询，确保 RLS 以目标用户身份生效。Tableau Server 版本 2025.3.4 完全支持，无降级风险。

3. **Mulan 内建"问数"前端模块**：在 Mulan 内部新建独立的"问数"页面，借鉴 Open WebUI 的 UX 理念（聊天框为核心、干净优雅），通过 Mulan 内部 Agent 接口与后端通信，用户只看到聊天框，不接触 Mulan 运维管理页面。

4. **聊天式问数**：用户用自然语言提问，Agent 调用已有的 `query-datasource` MCP 工具执行查询，返回数据结果和自然语言解释。

5. **RLS 保真**：查询全程使用目标用户 JWT，不回退到运维 PAT，确保数据隔离。

6. **基础错误反馈**：用户身份未映射、无权访问数据时，给出可理解的提示，不暴露系统内部信息。

## 四、非目标范围（Out of Scope）

| 排除项 | 说明 |
|--------|------|
| ETL 数据集成 | 项目级 Non-Goal，不做 |
| BI 可视化本身 | 不在"问数"模块中渲染图表（返回数据表格即可） |
| 多租户 SaaS | 项目级 Non-Goal |
| Mulan 管理功能暴露给普通用户 | "问数"模块不嵌入运维、连接配置等页面，与管理后台完全隔离 |
| 用户自助注册/修改身份映射 | 用户名由 AD/LDAP 自动透传，用户不能自行修改 |
| 跨数据源联合查询 | MVP 阶段只查 Tableau，不跨 DB/数仓 |
| 查询历史管理 | MVP 阶段不额外建查询日志 UI |
| Tableau 资产发布/修改 | 只读查询，不写入 Tableau |
| 接入外部 Open WebUI 实例 | 不部署或对接独立的 Open WebUI 实例，界面在 Mulan 内部自建 |

## 五、用户故事

**US-01**
As a 业务分析师，I want to 用自然语言在聊天框里问"上个月华东区的销售额是多少"，so that 我不需要打开 Tableau、不需要写 SQL，直接拿到答案。

**US-02**
As a 区域销售经理，I want to 问数时只看到我负责区域的数据，so that 我不会因为看到全国数据而产生误判，也不担心数据越权。

**US-03**
As a 财务分析师，I want to 在聊天窗口里追问"按月份拆分一下"，so that 我可以像和人对话一样逐步深入分析，不用每次重新开始。

**US-04**
As a 普通 BI 用户，I want to 当我问了一个没有数据权限的问题时，收到明确提示"您暂无该数据的访问权限"，so that 我知道是权限问题，而不是系统报错。

**US-05**
As a 普通 BI 用户，I want to 进入 Mulan "问数"页面后直接看到聊天框，不出现任何运维和配置界面，so that 我不会被无关功能干扰，页面干净易用。

**US-06**
As a BI 平台管理员，I want to 用户登录 Mulan 后系统自动以其 AD 账号透传 Tableau 身份，so that 无需手工维护用户名映射表，200 多个用户开箱即用。

**US-07**
As a BI 平台管理员，I want to 当某个用户的 AD 账号在 Tableau 中不存在对应用户时，收到告警，so that 我能及时处理异常账号，避免用户问数失败。

## 六、验收标准

**US-01 验收标准**
- AC-01-1：用户在 Open WebUI 输入自然语言问题，5 秒内收到 Mulan Agent 响应（不含查询执行时间）。
- AC-01-2：响应包含数据表格（或无结果说明）和一句自然语言解释（"以上数据来自 XXX 数据源"）。
- AC-01-3：后端日志可确认该次查询使用了目标用户的 JWT，而非运维 PAT。

**US-02 验收标准**
- AC-02-1：使用 Tableau 测试账号 A（仅有华东区权限），问"全国销售额"，返回结果仅含华东区数据，且无错误提示。
- AC-02-2：使用运维账号直接查同一问题，可返回全国数据——两次结果不同，证明 RLS 生效。
- AC-02-3：切换为账号 B（仅有华北区权限）重复测试，结果仅含华北区数据。

**US-03 验收标准**
- AC-03-1：在同一对话中追问"按月份拆分"，Agent 能理解上下文，不要求用户重述数据源和维度。
- AC-03-2：追问后的查询仍使用同一用户 JWT，RLS 不失效。

**US-04 验收标准**
- AC-04-1：用户查询无权访问的数据时，页面显示"您暂无该数据的访问权限，如需申请请联系 BI 团队"，不显示技术报错堆栈。
- AC-04-2：用户身份未映射时，显示"您的账号尚未完成 Tableau 身份绑定，请联系管理员"。

**US-05 验收标准**
- AC-05-1：普通 BI 用户登录 Mulan 并进入"问数"页面后，页面仅展示对话功能，不出现"连接管理"、"资产同步"、"用户管理"等运维入口。
- AC-05-2："问数"模块不挂载 Mulan 管理后台的任何路由。

**US-06 验收标准**
- AC-06-1：用户登录 Mulan 后，系统自动取其 AD/LDAP 用户名作为 Tableau 用户名签发 JWT，无需任何手工配置。
- AC-06-2：后端日志可确认每次 JWT 签发使用的用户名与 AD 登录名一致。

**US-07 验收标准**
- AC-07-1：系统在用户首次问数时，若 Tableau 返回"用户不存在"或"身份无效"错误，立即向管理员发送站内通知（或写入告警日志）。
- AC-07-2：管理后台提供"问数异常用户"列表，管理员可快速定位账号问题。

## 七、关键约束

### 7.1 RLS 保真（硬性约束）

- 每次查询必须签发携带目标用户 Tableau 用户名的 JWT，不得回退到运维 PAT。
- JWT 签发必须在 Mulan 后端完成，不得将 Connected App 密钥暴露给前端或 Open WebUI。
- JWT 有效期建议设置为单次查询生命周期（5~15 分钟），不复用。

### 7.2 "问数"前端模块边界

- "问数"模块作为 Mulan 内的独立前端路由（如 `/query`），与管理后台路由完全隔离。
- "问数"模块通过 Mulan 内部 Agent 接口与后端通信，不直接访问 Tableau。
- 普通 BI 用户只发放"问数"页面地址，不发放 Mulan 管理后台地址。
- 不部署、不对接任何独立的 Open WebUI 实例。

### 7.3 身份透传要求

- 身份来源：Mulan 与 Tableau 共用同一套 AD/LDAP，用户名一致，系统自动透传，无需额外映射表。
- JWT 签发使用当前登录用户的 AD 用户名，由 Mulan 后端完成，不暴露 Connected App 密钥给前端。
- 若 Tableau 端账号不存在（如已离职、未同步），查询失败并触发告警，不回退到运维 PAT。

## 八、成功指标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| RLS 保真率 | 100% | 所有查询均通过用户 JWT，零 PAT 回退 |
| 用户身份映射覆盖率 | ≥ 95%（上线 2 周内） | 200+ 用户中完成映射的比例 |
| 问数成功率 | ≥ 90% | 用户提问后拿到有效数据结果（含无权限提示）的比例，不含无效问题 |
| 首次响应时延 | ≤ 8 秒（P90） | 从用户发送到 Agent 开始回复 |
| 普通用户误入管理后台次数 | 0 | Open WebUI 不包含任何管理后台入口 |
| 管理员告警响应时间 | ≤ 1 个工作日 | 未映射用户告警到补全映射的时间 |
