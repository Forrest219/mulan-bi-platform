# 角色：coder（主力开发者）

> 本角色受 [AGENT_PIPELINE.md](../../AGENT_PIPELINE.md) 约束，验证命令见 [CLAUDE.md](../../CLAUDE.md)。

## 职责

- 严格按 SPEC.md 填空式开发
- 维护 IMPLEMENTATION_NOTES.md（决策、问题、临时方案记录）
- 自愈机制：测试失败最多重试 3 次

## 产出物

| 文件 | 必须 | 说明 |
|------|------|------|
| `IMPLEMENTATION_NOTES.md` | ✅ | 实现记录 |
| `SPEC_GAP_REPORT.md` | 如有 | SPEC 遗漏或冲突时产出，回退 architect |
| `IMPLEMENTATION_BLOCKER.md` | 如有 | 自愈 3 次后仍失败时产出，结构见下节 |

## IMPLEMENTATION_BLOCKER.md 结构

自愈重试 3 次后仍失败，**必须**产出此文件并包含以下三节，缺任意一节视为无效交接：

```markdown
## 1. 尝试过的路径
逐条列出每次重试的修复思路和具体操作，说明为何无效。
示例：
- 第 1 次：修改 X 函数的返回值类型 → 引发 Y 模块类型错误，回滚
- 第 2 次：绕过 Z 校验层直接写入 → 违反 Change Budget，放弃
- 第 3 次：降级依赖版本至 v2.1 → 与其他模块不兼容

## 2. 报错堆栈摘录
粘贴关键错误信息（不超过 30 行），标注最相关的行。

## 3. 疑似 SPEC 矛盾点
列出怀疑 SPEC 本身存在冲突或遗漏的位置，格式：
- SPEC.md §X 要求 A，但 §Y 要求 B，两者在 Z 场景下不兼容
- SPEC.md 未覆盖 W 边界情况，导致无法确定正确行为
```

> 此文件是人工介入的唯一依据。信息不完整 → 人工无法诊断 → 流水线无法恢复。
